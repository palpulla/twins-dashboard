# 3B Holdings Property Management Dashboard — Design Spec

**Date:** 2026-04-18
**Status:** Phase 1 design, ready for implementation planning
**Business:** 3B Holdings (real estate partnership, 3 partners)

## Problem

3B Holdings owns 4 rental properties (12 doors total), scaling toward 100+ over the next five years. Tenant-facing communication, rent collection, lease tracking, and tenant-side accounting are all handled in RentRedi — that works and is out of scope for this project.

What is scattered today is the **owner-side** management: mortgage terms and equity per property, repair history, vendor contacts, property documents (purchase docs, tax returns, insurance), and per-property expenses. Three partners manage remotely and need a shared source of truth so nobody has to ask "who paid that plumber?" or "when does the insurance renew?"

This project delivers a web dashboard that owns the back-office management of the portfolio.

## Goals

- Single source of truth for properties, mortgages, repairs, vendors, expenses, and documents.
- Three partners see the same numbers in real time.
- Every write is logged so the partnership has a clean audit trail.
- Designed for 100+ properties from day one; built to feel natural with today's 4.
- UI generated by Claude Design from this repo's spec + schema, so the build effort goes into modeling and docs rather than hand-crafted UI.

## Non-goals

- Tenant-facing anything (RentRedi owns it).
- Multi-company / white-label support — this is 3B Holdings only.
- Native iOS/Android app. Responsive web only.
- QuickBooks integration in v1.
- Rent-income tracking in v1 (deferred to Phase 2 via RentRedi API if available).
- Mortgage what-if scenarios in v1 (deferred to Phase 2).

## Users & access

Three partners, all equal admins with full read/write on everything. Invite-only. Supabase Auth with magic link (no password management).

A fourth user (or more) can be invited later by any admin — all invitees default to admin role in Phase 1. Role separation is not needed now.

Every write operation is recorded in an `activity_log` table with actor, action, entity, diff, and timestamp. Viewable per property in an Activity tab and globally in a Settings view.

## Scope

### Phase 1 — ship first

| Area | What's in Phase 1 |
|---|---|
| Properties & units | CRUD on properties (address, city/state/zip, purchase date, purchase price, current estimated value, manual value-updated timestamp, notes). CRUD on units under a property (label, bedrooms, bathrooms, sqft, notes). Single-family properties have one unit labeled "Main". |
| Mortgages | Per-property mortgage: lender, original principal, rate, term (months), start date, monthly payment, escrow-included flag, status (active/paid off/refinanced). Payment log with date, total amount, principal/interest/escrow split, extra-principal field. Auto-computed amortization schedule. Current principal balance is maintained by applying logged payments to the schedule. |
| Repairs | Per property (optionally scoped to a unit). Status flow: open → in progress → done (plus cancelled). Fields: title, description, vendor (optional), assignee (which partner is handling), opened date, completed date, cost, notes, linked invoice document. Marking a repair "done" auto-creates an `expenses` row in category `repairs`. |
| Vendors | Simple contact list: name, category (plumber / electrician / HVAC / handyman / landscaping / other), phone, email, notes. |
| Expenses | Per-property ledger. Categories: mortgage, insurance, tax, utilities, repairs, HOA, other. Amount, date, description, vendor (optional), receipt document (optional). CSV export for any date range. |
| Documents | Per-property library in Supabase Storage. Categories: purchase, tax, insurance, mortgage statement, inspection, receipt, other. Upload, download, delete, rename. |
| Deadlines | Per-property recurring or one-off deadlines (insurance renewal, property tax due, HOA dues, etc.). Title, due date, recurring cadence (none / monthly / annually), completed flag. Recurring deadlines auto-advance when marked complete. |
| Home dashboard | 4 widgets: Portfolio equity (net worth across all properties), Expenses this month, Open operations (open+in-progress repairs), Upcoming deadlines (insurance/tax renewal dates, etc.). Exact layout determined by Claude Design. |
| Audit log | Every create/update/delete recorded. Per-property Activity tab + global view in Settings. |

### Phase 2 — after real usage

- Mortgage what-if page (refi calculator, extra-principal payoff simulation, projected equity at 3/5/10 years).
- Year-end tax report (per property, per tax year) with categorized expenses, totals, PDF export.
- RentRedi income integration — if the API exposes rent payment data, pull it into the expenses/cash-flow views. Otherwise, CSV-upload of a monthly rent summary.

### Explicitly out of scope forever

- Tenant-facing portal, maintenance requests from tenants, rent collection.
- Multi-company / multi-tenant architecture.
- Native mobile apps.
- Marketing / public listing pages.

## Supabase schema

Eleven tables. Full SQL lives in `supabase/schema.sql` in the deliverable repo; this section is the conceptual shape.

### Core

**`properties`**
- `id` uuid PK, `address`, `city`, `state`, `zip`
- `purchase_date` date, `purchase_price` numeric
- `current_estimated_value` numeric, `value_updated_at` timestamptz
- `notes` text, `created_at`, `updated_at`

**`units`**
- `id` uuid PK, `property_id` FK → properties
- `label` text (e.g., "Unit 1", "Main"), `bedrooms` int, `bathrooms` numeric, `sqft` int, `notes` text
- `created_at`, `updated_at`

### Mortgages

**`mortgages`**
- `id` uuid PK, `property_id` FK → properties
- `lender` text, `original_principal` numeric, `interest_rate` numeric (APR as decimal), `term_months` int
- `start_date` date, `monthly_payment` numeric, `escrow_included` bool
- `status` text check in ('active', 'paid_off', 'refinanced')
- `notes` text, `created_at`, `updated_at`

**`mortgage_payments`**
- `id` uuid PK, `mortgage_id` FK → mortgages
- `payment_date` date, `amount` numeric
- `principal_portion` numeric, `interest_portion` numeric, `escrow_portion` numeric, `extra_principal` numeric (default 0)
- `source` text check in ('manual', 'recurring')
- `notes` text, `created_at`

The amortization schedule itself is computed (not stored) from the mortgage row. Current principal balance is computed as `original_principal - sum(principal_portion) - sum(extra_principal)` over all logged payments.

### Operations

**`vendors`**
- `id` uuid PK, `name`, `category` text, `phone`, `email`, `notes`
- `created_at`, `updated_at`

**`repairs`**
- `id` uuid PK, `property_id` FK → properties, `unit_id` FK → units (nullable)
- `title`, `description`
- `status` text check in ('open', 'in_progress', 'done', 'cancelled')
- `opened_date` date, `completed_date` date (nullable)
- `vendor_id` FK → vendors (nullable)
- `assigned_to` FK → partners (nullable)
- `cost` numeric (nullable until done)
- `invoice_document_id` FK → documents (nullable)
- `notes` text, `created_at`, `updated_at`

**`expenses`**
- `id` uuid PK, `property_id` FK → properties
- `expense_date` date, `amount` numeric
- `category` text check in ('mortgage', 'insurance', 'tax', 'utilities', 'repairs', 'hoa', 'other')
- `description` text
- `vendor_id` FK → vendors (nullable)
- `receipt_document_id` FK → documents (nullable)
- `source_repair_id` FK → repairs (nullable — populated when auto-created from a completed repair)
- `notes` text, `created_at`, `updated_at`

### Documents

**`documents`**
- `id` uuid PK, `property_id` FK → properties
- `title`, `category` text check in ('purchase', 'tax', 'insurance', 'mortgage_statement', 'inspection', 'receipt', 'other')
- `storage_path` text (path in the `property-documents` bucket)
- `file_size` bigint, `mime_type` text
- `uploaded_by` FK → partners, `uploaded_at` timestamptz
- `notes` text

One Supabase Storage bucket: `property-documents`. Files stored under `{property_id}/{document_id}/{original_filename}`.

### Access & audit

**`partners`**
- `id` uuid PK (= `auth.users.id`), `name`, `email`, `role` text check in ('admin')
- `created_at`

**`activity_log`**
- `id` uuid PK, `actor_id` FK → partners
- `action` text check in ('create', 'update', 'delete')
- `entity_type` text, `entity_id` uuid
- `property_id` uuid (denormalized, for filtering; nullable for entities not tied to a property, e.g., vendor changes)
- `diff` jsonb (before/after snapshot)
- `timestamp` timestamptz

### Deadlines

**`deadlines`**
- `id` uuid PK, `property_id` FK → properties
- `title` text (e.g., "Insurance renewal", "Property tax due")
- `due_date` date, `recurring` text check in ('none', 'monthly', 'annually')
- `completed` bool, `notes` text
- `created_at`, `updated_at`

Drives the "Upcoming deadlines" widget. Recurring deadlines auto-advance when marked complete.

## Row-Level Security

- `auth.uid()` must exist in `partners` for any table access.
- All 11 tables: SELECT/INSERT/UPDATE/DELETE allowed to any row if the caller is in `partners`.
- `activity_log`: SELECT allowed to all partners. INSERT only via server-side trigger (not client). No UPDATE/DELETE ever.

Write triggers on every CRUD table insert a matching `activity_log` row automatically — actor comes from `auth.uid()`, diff is computed from OLD/NEW. This removes the burden of the client remembering to log.

## Screens

**Global shell:** left sidebar with nav links, top bar with current user + logout, main content area.

**Sidebar:** Home · Properties · Repairs · Vendors · Documents · Settings

1. **Home** — 4 widgets (Portfolio equity, Expenses this month, Open operations, Upcoming deadlines). Each widget links to its full page for detail.
2. **Properties** — table view. Columns: address, # units, current value, equity %, open repairs count, expenses YTD. Sortable, searchable by address. Row click → property detail.
3. **Property Detail** — tabbed. Tabs:
   - **Overview** — address, value, equity snapshot, quick stats, recent activity feed.
   - **Units** — list/CRUD.
   - **Mortgage** — terms summary, payment log, amortization table, "Log payment" action.
   - **Repairs** — filtered list of repairs for this property.
   - **Expenses** — filtered ledger for this property, CSV export button.
   - **Documents** — file library for this property.
   - **Deadlines** — upcoming + completed deadlines for this property.
   - **Activity** — audit log filtered to this property.
4. **Repairs** (global) — all open + in-progress across portfolio by default, filter to see completed. Columns: property, unit, title, status, vendor, opened, cost.
5. **Vendors** — contact list, filterable by category.
6. **Documents** (global) — all documents across the portfolio with property + category filters. Same data as per-property tabs, just the cross-portfolio view.
7. **Settings** — partners list, invite new partner (email → magic link), global activity log, profile, sign out.

**Auth screens:**
- **Login** — email field → magic link sent.
- **Accept invite** — lands here from magic link on first sign-in, completes name field if missing.

## Deployment model: Claude Design

The deliverable is a GitHub-ready repo folder that contains everything Claude Design needs to generate the UI. The human workflow:

1. The operator (or a partner) creates a new empty GitHub repo, e.g. `3b-holdings-dashboard`.
2. Push the deliverable folder to that repo.
3. Point Claude Design (https://claude.ai/design/) at the repo.
4. Claude Design reads the spec and schema, generates UI pages and components.
5. The operator reviews the generated UI, iterates via Claude Design prompts if needed.
6. Deploy per Claude Design's output path.

Separately, the operator sets up a Supabase project and runs `supabase/schema.sql` + `supabase/seed.sql` against it. The repo's `.env.example` documents the two env vars (`VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`) that Claude Design's output must consume.

## Style direction

Minimal opinion from us. `05-style-guide.md` will say:

> No strong design preference. Target a modern, professional property-management feel: legible, data-friendly, desktop-first but responsive down to mobile. Typography should make numbers easy to scan. Prefer clean neutrals with a single accent color. Avoid heavy decoration; this is a working tool, not a marketing site.

This leaves Claude Design real freedom while constraining tone.

## Open questions for implementation

These are not blocking the spec but will be resolved while writing the plan:

- Where exactly the repo folder lives on disk before push to GitHub.
- Whether initial data entry for the 4 current properties happens via manual UI entry (after generation), a one-time SQL seed authored from the operator's uploaded documents, or both. Proposed default: the operator uploads the documents, I extract the data and write a one-off seed SQL as a follow-on task after the repo is generated.
- Whether the partner invite flow in Phase 1 is a manual Supabase dashboard operation (add user by email) vs. an in-app invite button. Proposed default: manual via Supabase dashboard for v1; in-app invite in Phase 2.

## Success criteria

- The operator can push the folder to GitHub and Claude Design produces a working dashboard on first try, or with at most minor iteration.
- All 3 partners can sign in via magic link and see the same data.
- Adding a property, logging a mortgage payment, logging a repair from open → done, uploading a document, and viewing the dashboard all work end-to-end against a real Supabase instance.
- Every write is visible in the Activity view with the correct actor.
- The 4 home-page widgets reflect real data correctly for the 4 seeded properties.
